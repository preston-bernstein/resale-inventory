## Changes made

- **Fixed a silent data-corruption bug found independently by two agents**: the migration's `INSERT INTO items_v2 SELECT * FROM items` was positional and didn't match the plan's own column order (`tenant_id` was sketched second, but the real live table has it last via an `ALTER TABLE ADD COLUMN`). A bare `SELECT *` here would have silently scrambled every existing row's data (e.g. `category`'s value landing in `tenant_id`) while still passing the only test the original steps.md specified (a row-count check). Fixed to an explicit column list on both sides of the INSERT, reordered `items_v2` to match the real live column order, and added a new Step 0 that verifies the migration's assumed column list against the actual desktop DB's `.schema items` output before it ever runs.
- **Fixed a real, currently-true security gap**: `PATCH /api/items/[id]`'s platform-field validation today accepts any `string[]` with zero check against category-appropriate platforms — meaning the planned category-filtered picker (AC15/16) would have been purely a UI decoration with no server-side enforcement; a raw PATCH call could still set a book item's platforms to `['swappa']`. Added a single `PLATFORM_CATEGORY_SUPPORT` map used by the picker, every connector's rejection logic, AND a new PATCH-side 422 check — one source of truth instead of the originally-planned 10 separate hand-written checks.
- **Fixed a silent-miscategorization hazard from non-exhaustive branching**: `POST /api/items`, `fetchDetails`, and the PATCH allowlist logic were all `if/return` chains with no compiler-enforced exhaustiveness (unlike `lib/types.ts`'s `Item` union, which does force a compile error). If the constants/types widening shipped before the route-handler widening — plausible, since many steps were marked parallelizable — an electronics POST would have silently fallen through and been stored as `category: 'clothing'`. Fixed by requiring exhaustive `switch`/`never`-check restructuring at these three call sites, matching the pattern already used correctly elsewhere in the codebase.
- **Loosened `device_type`'s one-way-door CHECK**: the original plan hard-restricted `device_type` to exactly `'laptop'` via a DB CHECK, which would have required a full table rebuild the moment a second device type (phone, tablet) is ever added — despite the category itself being named "electronics," not "laptops." Changed to app-layer-validated-only (mirroring how `SUPPORTED_PLATFORMS` already avoids a DB CHECK for exactly this reason), so a future device type needs zero migration for this column.
- **Split 3 oversized/undersized steps and fixed 5 dependency-graph errors**: the DB migration (mixed high-risk CHECK-rebuild with low-risk additive-table work), the Swappa connector (5 methods + session plumbing in one step), and the platform-picker rework (bundled with unrelated additive rendering, and silently assumed pre-existing `item_platforms` data was clean) were all split for independent verifiability; `lib/constants.ts`/`lib/types.ts` were freed from an unnecessary serialization chain onto the migration.
- **Widened the electronics PATCH allowlist**: originally only `condition`/battery fields were correctable after creation — `brand`/`model`/specs were locked at creation time with no correction path except delete-and-recreate. Now all `electronics_details` fields are patchable pre-sale, matching the realistic case of a typo'd spec field.
- **Added a data-integrity trigger and index gaps found by the Data Model Critic**: a category-matching trigger on `electronics_details` (previously only tenant-matching existed — nothing stopped a satellite row from attaching to a non-electronics item), a `(tenant_id, category)` composite index for the dashboard's actual query shape, a `model` index to match the existing `brand` index, and the missing UUID-format CHECK on `items.id` that every migration since `007` has carried but this rebuild's sketch omitted.

## Critiques rejected

- **Config-map generalization across dashboard/CSV/5 connector field-mappers** (Design Devil's Advocate) — a real DRY concern, but the existing codebase already hand-copies book/clothing branches the same way; consolidating now would be a refactor of already-shipped code well beyond "add a third category," not a fix this feature needs to ship. Deferred as tech debt (see Open questions).
- **Length limits on `brand`/`model`/`processor`, upper bounds on `ram_gb`/`storage_gb`** (Security Auditor) — no precedent for TEXT length CHECKs anywhere in this schema (`clothing_details.brand` has none either); inventing a business threshold not requested anywhere in context would violate the spec's own rule against fabricating numeric limits.
- **SQL injection on brand/model search, Swappa credential-at-rest encryption, Playwright trace credential leakage** (Security Auditor) — all three are already fully covered by existing repo-wide conventions (parameterized `better-sqlite3` prepared statements throughout, the existing encrypted credential table constraint, and the existing "no credential in log/error/trace" NFR that already applies to all 8 connectors and now extends to Swappa by the same rule).
- **CSV import encoding validation, CSV endpoint auth** (Security Auditor) — hypothetical at irrelevant scale for a single-operator local app, and the auth question is a pre-existing app-wide concern this feature doesn't change.
- **`ELECTRONICS_CONDITIONS`/`SWAPPA_ACTION_INTERVAL_MS` being "leaked" into plan.md** (Requirements Auditor) — these were explicit TBD placeholders in requirements.md; plan.md resolving them is the plan doing its job, not a defect.
- **Exact form-field enumeration for `AddElectronicsForm`/`ElectronicsDetailRows` belongs in requirements.md** (Requirements Auditor) — requirements.md is already at the correct altitude ("mirrors book/clothing UX"); that level of UI-field detail correctly lives in plan.md/steps.md.

## Open questions requiring human input

- **Laptop-only scope vs. Swappa's real user base**: Swappa (this feature's flagship new connector) is historically known primarily for phone resale, not laptops. Shipping a laptop-only electronics category alongside it is a product-fit mismatch nobody has resolved — worth deciding whether phones should be pulled into v1, or whether a different platform would be a better flagship for a laptop-only launch.
- **`ELECTRONICS_CONDITIONS` vocabulary** (`New`/`Excellent`/`Good`/`Fair`/`For Parts`) is the architect's best guess, not verified against Swappa's actual listing-condition dropdown. Cheap to change now; expensive (another full CHECK-rebuild) once real rows exist — worth a 10-minute check against Swappa's real form before this ships.
- **Swappa's real submission flow may not be "instant"** like the fashion-resale connectors (Grailed/Poshmark/Mercari) this connector's shape was borrowed from — Swappa has historically required device/IMEI verification and listing review. Worth confirming before trusting `createListing`'s synchronous-return assumption, ideally with real account access.
- **Category-branch consolidation** (dashboard/CSV/connector field-mapping) was flagged and deliberately deferred as tech debt rather than fixed here — worth scheduling if a 4th category ever becomes likely, since the cost of consolidating grows with each category added by hand.
