# Plan: Playwright QA Harness

## Approach
`tests/e2e/` already has 11 spec files, a working multi-tenant `setup` project (`auth.setup.ts` → `STORAGE_STATE_PATH`), and locator/flow helpers (`helpers.ts`) — a direct read of every file (not a grep, not an assumption) shows all five simple flows and three of four complex flows already have real coverage, including FR3/AC3's marketplace-connector flow: `tests/e2e/connections-flow.spec.ts` already drives Depop's full consent → credential entry → masked confirmation → first-win panel end-to-end in dry-run mode with no live network calls — confirmed by reading the file. This plan does not add a new connector spec; it only needs a verification pass during build-out (run the spec, confirm it currently passes). The two real gaps are (1) no spec proves session persistence across navigation/reload using the suite's actual shared-tenant model, and (2) the CSV round-trip test only spot-checks a handful of columns rather than proving no field is lost. The approach is therefore additive and corrective, not a rebuild:

- Add a session-persistence test to `auth-pages.spec.ts` that reuses the **default already-authenticated** `storageState` every other spec in the `chromium` project uses — it does **not** override `storageState` to empty and re-run signup through the UI. Doing that would create a brand-new tenant on every single test run in a database that is never wiped between runs, silently violating the "one E2E tenant" invariant `playwright.config.ts` documents. See Integration points for the exact mechanism.
- Extend `csv-export-import.spec.ts` with a full-field round trip built from its own new row(s) only (never by exporting-then-reimporting the whole existing dataset), and fix a real, previously-unflagged bug in that same file while we're in it: its `suffix` constant (`tests/e2e/csv-export-import.spec.ts:27`) is computed once at module load, outside any test — confirmed by reading the file. That breaks FR11/AC2's `--repeat-each=3` requirement, since Playwright reuses the same module-level suffix across all 3 repeats and the 2nd/3rd repeat collide on title/ISBN.
- Extract the book-item-creation boilerplate duplicated across five spec files into `helpers.ts` as a category-specific `createBookItem(page, { title, author?, cost?, date? })`, mirroring the already-established `createClothingItem(page, brand)` pattern. Two separate, category-specific helpers stay separate — a generic `createItem(page, category, fields)` was considered and rejected as over-engineering beyond this feature's scope (it would also force rewriting `createClothingItem` and all its call sites, which isn't the duplication being fixed here).
- `phone-handoff.spec.ts` and `photo-upload.spec.ts` each already declare their **own local** `async function createBookItem(page, title)` (confirmed by reading both files — `phone-handoff.spec.ts` additionally reimplements `fieldWrapper`'s label→xpath→input chain inline instead of calling it). Importing the shared helper means **deleting** both local declarations, not renaming them on import — a rename-on-import would silently leave the old, duplicated logic running with no compile error to catch it.
- `dashboard.spec.ts` and `search-filter.spec.ts` do **not** have simple 1:1-swappable inline book creation — each declares a combined `createBookAndClothing(page)` that creates one book **and** one clothing item together (confirmed by reading both files). This is a bigger, distinct edit: decompose each into two calls — `helpers.createBookItem` + `helpers.createClothingItem` — with the caller generating and threading its own unique title/brand via `uniqueSuffix()`, matching the pattern both files already use elsewhere.
- Any application defect the new/extended specs surface is triaged per FR10: a data-integrity/correctness defect gets fixed in this branch with a Playwright regression assertion, plus a Vitest regression test too if the fix lands in code with existing Vitest unit coverage; anything else surfaced gets named in the delivery report rather than fixed silently, keeping scope bounded.

## Architecture
No new components. Existing flow, confirmed by reading the config:

```
npm run test:e2e
   -> playwright.config.ts
        -> webServer: `next dev --turbopack -p 3100`
             env: BOOKSELLER_DB_PATH / BOOKSELLER_PHOTOS_PATH / BOOKSELLER_CREDENTIAL_KEY_PATH
                  all -> .playwright-scratch/ (never data/)
        -> project "setup" (auth.setup.ts): signs up one throwaway tenant via
           request.post('/api/auth/signup'), saves cookie to STORAGE_STATE_PATH
        -> project "chromium" (dependencies: ['setup']): every *.spec.ts except
           auth.setup.ts, loads STORAGE_STATE_PATH as storageState
             -> tests/e2e/*.spec.ts drive real browser pages against the
                webServer, asserting via getByRole/getByText/helpers.ts locators
        -> reporter: 'list' locally, [html, github] on CI -> playwright-report/
```

New/modified specs slot into the existing `chromium` project; nothing about the dependency graph, webServer, or reporter changes. The new session-persistence test in `auth-pages.spec.ts` is the one exception worth diagramming: that file currently applies `test.use({ storageState: { cookies: [], origins: [] } })` at module scope for its (deliberately unauthenticated) signup/login tests. The new test lives in its own nested `test.describe` with its own `test.use({ storageState: STORAGE_STATE_PATH })`, which overrides the file-level empty override for that scope only — Playwright resolves `test.use` innermost-wins, so the new test gets the same default authenticated session as every spec in `chromium`, without touching the outer file's existing tests. The addition to *process* (not architecture) is running the new `npm run test:e2e:flaky-check` script (`playwright test --project=chromium --repeat-each=3`, see API/interface contract) against each new/modified spec file during authoring (FR11), and a final full `npm run test:e2e` x3 at the end of build-out (AC2).

## Data model
No data model changes. Verified: this feature is test infra plus, potentially, application bug fixes surfaced while writing/running the new specs — no migration, schema, or table change is anticipated by anything in requirements.md, and none of the gaps found during the audit (session-persistence coverage, CSV round-trip depth, helper duplication) implicate the data model. Per FR10's triage rule (see Approach), if a defect fix discovered during build-out turns out to require a schema change, that's a scope deviation to flag explicitly at that point, not something planned here.

## API / interface contract
One new `package.json` script: `"test:e2e:flaky-check": "playwright test --project=chromium --repeat-each=3"`. This gives the FR11/AC2 determinism check a permanent, discoverable form instead of a command someone has to remember to type manually with no lasting trace after this PR — used during authoring of each new/modified spec (FR11) and, run against the full suite, as the AC2 gate at the end of build-out. No other new routes, CLI flags, or npm scripts; `npm run test:e2e` remains CI's unchanged entry point.

## Integration points
- `tests/e2e/helpers.ts` — add:
  - `createBookItem(page: Page, book: { title: string; author?: string; cost?: string; date?: string }): Promise<void>`, mirroring `createClothingItem(page, brand)`'s shape (navigate to `/inventory/new`, click the `Book` tab, fill via `inputByLabel`, submit, land back on `/inventory`). The object-param shape mirrors the local `NewBook` interface `book-flow.spec.ts` already declares (`title`, `author`, `cost`, `date`, all currently required there). `title` has no default — every call site must pass it, matching every existing call site today. Optional fields get generated defaults so an omitted field never silently collides with another spec's fixture in the shared, never-wiped DB: `author` defaults to `` `E2E Author ${uniqueSuffix()}` `` (a free-text field that participates in substring search assertions elsewhere in the suite). `cost` and `date` are not identity/search fields — no existing or planned spec asserts uniqueness or an exact readback of either — so they get fixed, valid defaults (`'9.99'`, `'2026-01-15'`) rather than a `uniqueSuffix()`-derived value, which would just produce type-invalid noise (a suffix isn't a valid currency amount or date) for no collision-avoidance benefit.
  - `createClothingItem` is **not** modified and is **not** generalized into a shared `createItem` — rejected as over-engineering (see Approach).
- `tests/e2e/book-flow.spec.ts` — its local `addBook(page, book: NewBook)` always passes all four fields explicitly already (no field is ever omitted at either call site). Swap it for the new `helpers.createBookItem`, delete the local `addBook` function and `NewBook` interface; call sites are otherwise unchanged (same explicit `{title, author, cost, date}` object literals).
- `tests/e2e/phone-handoff.spec.ts` and `tests/e2e/photo-upload.spec.ts` — both declare a local `async function createBookItem(page, title)` (confirmed identical pattern in both files; `phone-handoff.spec.ts`'s version additionally reimplements the `label` → `xpath=..` → `input` chain inline instead of calling `inputByLabel`/`fieldWrapper`). **Delete both local declarations** and import `createBookItem` from `helpers.ts` instead. Both files only ever call it with a title (`author`/`cost`/`date` were previously hardcoded to file-specific placeholder text, e.g. `'E2E Phone Handoff Test Author'` / `'E2E Photo Test Author'`) — since neither file asserts on author/cost/date values anywhere, both call sites become `helpers.createBookItem(page, { title })`, relying on the new helper's defaults for the rest.
- `tests/e2e/dashboard.spec.ts` and `tests/e2e/search-filter.spec.ts` — **not** a 1:1 swap. Each currently declares its own combined `createBookAndClothing(page)` (confirmed by reading both files) that creates one book and one clothing item together, using a locally-generated `suffix` and file-specific costs (dashboard: book `12.34`, clothing `56.78`; search-filter: book `9.99`, clothing `19.99`) and explicit `Condition` selections (`Good` for books, `EUC` for clothing) that happen to equal each form's own default. On direct read, **neither file actually asserts on those specific cost values** — `dashboard.spec.ts` only asserts the cost card matches a `$[\d,]+\.\d{2}` format, and `search-filter.spec.ts` never reads cost back at all; both files search/assert by title or brand only. So no call site here is forced by an existing assertion to pass an explicit cost — but this refactor still threads the current literal values through explicitly (rather than falling back to the new helper's defaults) purely to keep the diff behavior-preserving and avoid an unrelated fixture-data change riding along with a refactor PR. Concretely: replace `createBookAndClothing(page)` with two calls — `helpers.createBookItem(page, { title: bookTitle, author: 'Test Author', cost: '12.34' /* or 9.99 */, date: '2026-01-01' })` then `helpers.createClothingItem(page, clothingBrand)` — with the caller generating `bookTitle`/`clothingBrand` via `uniqueSuffix()` itself (the pattern both files already use elsewhere, e.g. `search-filter.spec.ts`'s existing `createBookAndClothing` return value). One real behavior change to flag explicitly: `createClothingItem` is unmodified (fixed cost `'25.00'`, date `'2026-02-01'`) — so the clothing row's cost/date silently changes from each file's current bespoke value (`56.78`/`2026-01-01` for dashboard, `19.99`/`2026-01-01` for search-filter) to `createClothingItem`'s fixed values. Confirmed harmless (no assertion in either file depends on clothing cost/date), but this is exactly the kind of implicit consequence worth stating up front rather than discovering mid-refactor. Also: dropping the explicit `Condition` selects is safe only because they already equal the AddBookForm/AddClothingItemForm defaults (`Good`/`EUC`) that `helpers.createBookItem`/`createClothingItem` leave in place — `dashboard.spec.ts`'s `By Condition` section assertions for `'Good'`/`'EUC'` keep passing because the default happens to match, not because the helper explicitly sets it.
- `tests/e2e/auth-pages.spec.ts` — add one new test in its own `test.describe('Session persistence (authenticated)', ...)`, with `test.use({ storageState: STORAGE_STATE_PATH })` (imported from `./storageStatePath`) inside that describe to override the file's module-level empty-`storageState` for this scope only. The test: create an item via `helpers.createBookItem(page, { title })` (a unique, `uniqueSuffix()`-based title), `page.goto` to a different route (e.g. `/dashboard`), `page.reload()`, then assert (a) the tenant is still authenticated — no redirect to `/login`, and the file's existing local `getSessionCookie(page)` helper still returns a truthy cookie — and (b) navigating back to `/inventory` and finding the item by title (`helpers.findItemCard`) still shows it. **No new signup, no `storageState` override to empty, no re-driving signup through the UI** — this closes the FR3 "session persists across navigation and reload" requirement without creating a new tenant on every run. Two locator-convention notes for whoever writes this test: (1) this file's *other* tests use real `page.getByLabel(...)` because the signup/login pages have genuine `<label htmlFor>`/`id` associations (confirmed in this file's own header comment and its usage) — that does **not** apply to the item-creation step here, since `helpers.ts`'s own header comment confirms the item form has no such association; any item-form locator in the new test must go through `helpers.inputByLabel`/`fieldWrapper`, or it will silently resolve to nothing. (2) `getSessionCookie` is confirmed spec-local to this file already (declared at `auth-pages.spec.ts:25`, not exported from `helpers.ts`) — the new test reuses it as-is; no code needs to move.
- `tests/e2e/csv-export-import.spec.ts` — two edits to the same file:
  1. **Bug fix**: move `suffix` (currently `const suffix = ...` at module scope, `tests/e2e/csv-export-import.spec.ts:27`, shared by every test in the file) to per-test scope — generate it fresh inside each test body (or call `uniqueSuffix()` directly at each use site) so `--repeat-each=3` doesn't reuse the same title/ISBN across all 3 repeats and collide on the 2nd/3rd.
  2. **New test**: a full-field CSV round trip. It builds its own new row(s) via the file's existing `buildCsv`/`buildCsvRow` pattern (direct CSV construction from `CSV_HEADERS`) — it does **not** `GET /api/export` the full existing dataset and reimport all of it. Reimporting everything would risk duplicating every prior row already in the shared, never-wiped DB if `/api/import` does plain inserts rather than upserts, which would break `dashboard.spec.ts`'s and `search-filter.spec.ts`'s aggregate-count/search assertions for reasons unrelated to what this new test checks. The fixture must include at least one non-round-dollar monetary value (e.g. `19.99`, or a value that forces a gross-profit subtraction) — the plan's own risk section (below) predicts a money-rounding defect as the most likely thing this test catches, and an all-round-dollar fixture wouldn't exercise that path at all. The test then `POST /api/import`s the CSV, `GET /api/export`s, and asserts every field the app is expected to preserve round-trips byte-for-byte, excluding `id`/timestamps/sale-only columns the import route deliberately ignores (per its own documented FR22/AC12 behavior).
- `tests/e2e/connections-flow.spec.ts` — **no new spec needed.** Confirmed by reading the first 140 lines: it already drives Depop's full consent → credential entry → masked confirmation → first-win panel flow end-to-end in dry-run mode, with generated (`uniqueSuffix()`-based) fake credentials, never a real marketplace call. FR3/AC3's marketplace-connector requirement is already satisfied. The only build-out task here is a verification pass — run this spec, confirm it currently passes — not new test authorship.
- `playwright.config.ts` — audited per FR8 by **direct read**, not a keyword grep (a grep can miss an assumption phrased without the exact searched terms). Already confirmed clean: its `fullyParallel: false` comment correctly says "tests share one server/DB (and, since Task 22, one E2E tenant)", not a pre-auth assumption. The remaining files under `tests/e2e/` still need the same direct-read treatment as part of build-out (not just the grep this plan originally scoped) before FR8/AC9 can be considered satisfied.
- Application code under `app/`, `lib/`, `components/` — no changes planned up front. Per FR10's triage rule (Approach), any data-integrity/correctness defect the new/modified specs surface during authoring gets fixed here with a Playwright regression assertion (plus a Vitest regression test if the touched code has existing Vitest unit coverage); anything else surfaced is named in the delivery report, not fixed silently. The specific files, if any, are unknown until the specs run.
- `package.json` — add the `test:e2e:flaky-check` script described in API/interface contract above.
- Test-data hygiene (documentation-only, no code change): any fake credentials used anywhere in this feature's build-out must be generated/unique (`uniqueSuffix()`-based emails/identifiers, the dry-run fake-credential pattern `connections-flow.spec.ts` already uses) — never hardcoded literals. This is already the repo's convention; note it applies more narrowly than it might first appear once the finding above (session-persistence test no longer signs up or logs in — it reuses the default `storageState`) is applied: the only place this feature touches credentials at all is the `connections-flow.spec.ts` verification pass, which already follows the convention in existing code with no edits needed.

## Technology choices
No new libraries or frameworks. Playwright's built-in `--repeat-each` CLI flag satisfies the FR11/AC2 determinism requirement (not a new dependency or a hand-rolled loop script), now wrapped in the new `test:e2e:flaky-check` package.json script for discoverability, exercising the exact same test through the exact same `chromium` project (with `setup` still resolving as a one-time dependency).

## Risk areas
- **Unknown defect surface area.** FR10/AC6 commit to fixing data-integrity/correctness defects the new specs find, per the triage rule in Approach — anything else gets named in the delivery report instead, keeping the work bounded. Nothing here can size the fix work in advance. The CSV full-round-trip test is the one most likely to expose a real data-loss or rounding bug (e.g. a clothing measurement field or a money value not surviving import unchanged), since the current spec never checked most of the 31 export columns and never included a non-round-dollar amount — this plan's fixture design (Integration points, above) deliberately includes one to actually exercise that path.
- **Timing-sensitive existing flows.** `phone-handoff.spec.ts` polls with 10s timeouts and `connections-flow.spec.ts`'s suspend/reactivate test writes directly to the scratch SQLite file from a second Node process sharing `BOOKSELLER_DB_PATH`. Neither is being touched functionally by this plan (`phone-handoff.spec.ts` only gets its local `createBookItem` deleted in favor of the shared helper), but the FR11 `--repeat-each=3` and AC2 full-suite-x3 determinism checks will surface any latent flakiness in them — and per the out-of-scope note, fixing a genuine flake means finding the real cause, not padding a timeout.
- **`presale-tour.spec.ts`'s existing `page.waitForTimeout` calls are out of scope.** FR5/AC5 bind "new or modified spec files" only; this plan does not touch that file, so its six existing fixed sleeps are left as-is. If a defect fix elsewhere forces an edit to that file, its sleeps would then need addressing too — flagging this now so it isn't missed at that point.
- **Scratch-DB reuse across repeated runs.** The scratch DB is never wiped between runs (by design, matching the existing suite's pattern), so every new test must keep using the established `uniqueSuffix()` convention to stay collision-free across the FR11 `--repeat-each=3` and AC2 three-consecutive-full-suite-run checks — this is exactly the bug class the `csv-export-import.spec.ts` `suffix` fix above closes. Unbounded growth of the scratch DB across many repeated *local* runs is a real but low-severity side effect: in CI it's bounded by the ephemeral runner's lifetime (fresh checkout every run), and for local dev a developer can freely `rm -rf .playwright-scratch` at any time (already true today; this feature adds no new cleanup mechanism, none is needed).
- **FR8/AC9 audit confidence.** The audit method is a direct read of `playwright.config.ts` and every file in `tests/e2e/`, not a keyword grep alone (a grep can miss an assumption phrased without the exact searched terms). `playwright.config.ts` has already been confirmed clean this way. The remaining spec files still need the same direct-read treatment during build-out before FR8/AC9 can be marked satisfied.
