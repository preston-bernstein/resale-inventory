# Requirements: Interactive Presale Tutorial with Motion UX

## Problem statement
First-time sellers currently learn the add-item/prep workflow by reading a static 17-step wall of text on `/playbook` (`app/playbook/page.tsx`), disconnected from the actual add-item form (`components/AddClothingForm.tsx`, `components/AddBookForm.tsx`) they then have to go operate from memory. There is no in-context guidance anywhere in the app — zero tooltips, modals, or coach marks exist in the codebase today. A seller adding their first item has to context-switch between the playbook page and the form, self-translate prose into actions, and re-derive which form field corresponds to which playbook step. This matters now because the app is about to grow a second entry point for this same guidance (a dashboard tutorial trigger) and the underlying content (`docs/clothing-resale-research.md` Section 7) already exists in two forms (17-step array in the playbook page, prose in the research doc) — a third static copy would compound the duplication this feature is explicitly meant to end.

## Users / stakeholders
- Primary user: the reseller (the app's sole operator/admin) adding a clothing or book item for the first time, or returning to refresh their memory on the workflow.
- Secondary: the same user in an experienced state, who must be able to bypass the tutorial entirely without friction.
- Maintainer (same person, in a dev capacity): must keep step content in one authoritative source as the workflow evolves.

## Functional requirements
1. The system shall present an optional, user-initiated tutorial ("presale tour") that walks through the real add-item workflow using coach marks anchored to actual elements in `AddClothingForm.tsx` / `AddBookForm.tsx` (or the page hosting them), not a standalone slideshow of prose.
2. The system shall expose exactly one entry point to start the tour for this iteration — a button in the add-item flow (`app/inventory/new/page.tsx`) — and the tour shall not start automatically without this explicit user action.
3. The tour shall consist of 5-7 steps, each tied to one real UI element (form field, button, or section) relevant to the add-item / prep workflow, sourced at runtime exclusively from the shared extracted array `SELLER_WORKFLOW_STEPS`. `docs/clothing-resale-research.md` Section 7 is background provenance for that content only and is not a second required runtime source. There are two distinct 6-step tour variants derived from `SELLER_WORKFLOW_STEPS` — `CLOTHING_TOUR_STEPS` for the clothing workflow and `BOOK_TOUR_STEPS` for the book workflow — each independently satisfying the 5-7 step count.
4. The system shall render each tour step as a tooltip/popover anchored to its target element, positioned so the target remains visible and not obscured by the tooltip itself.
5. The system shall automatically scroll the anchored target element into view if it is outside the current viewport when a tour step is shown.
6. The system shall provide a "Next" control that advances to the following step and a "Back" control that returns to the previous step. Back shall be hidden (not merely disabled) on step 1. Next's label shall change to "Finish" on the last step. Skip shall be visible on every step (see requirement 7).
7. The system shall provide a persistently visible "Skip"/"Close" control on every step that immediately and unconditionally ends the tour, tearing down all tour UI (tooltip, modal, overlay) in a single, non-animated step with no partial/stuck tour state left behind. This immediate teardown is intentional and does not need to satisfy the animated-exit-transition requirements of requirements 12 and 13, which apply only to normal step-advance (Next/Back) and to the completion modal's own close button.
8. The system shall preserve the user's in-progress form input (any field already filled in `AddClothingForm.tsx` / `AddBookForm.tsx`) across tour start, step navigation, and tour dismissal — the tour shall never clear or overwrite form field values.
9. The system shall persist, in `localStorage`, whether the user has completed or explicitly dismissed the tour at least once, tracked independently per item category (book vs. clothing) — completing or dismissing the tour for one category shall not suppress the entry point's ability to present the other category's tour as not-yet-seen — and shall not force-display the tour automatically on subsequent visits for a category already completed/dismissed.
10. The system shall allow a user who has previously completed/dismissed the tour to manually re-launch it at any time via the same entry point from requirement 2.
11. The system shall render a completion state (final step reached) that is visually distinct from a skipped/dismissed state, and shall return focus to the add-item form in both cases. The completion state is satisfied by displaying the `TourCompletionModal`; the skipped/dismissed state is satisfied by the immediate teardown of requirement 7, with no modal shown at all. These two outcomes are inherently visually distinct and require no additional dedicated "skipped state" UI.
12. Any modal introduced by this feature (e.g., the `TourCompletionModal`) shall support an explicit enter transition on open. When closed via normal step-advance (Next/Back reaching or leaving the modal) or via the completion modal's own close button, it shall play an explicit exit transition and shall not unmount instantly. This exit-transition requirement does not apply when the modal is torn down via Skip/Close per requirement 7, which is an intentional immediate, non-animated teardown.
13. Any tooltip introduced by this feature shall fade in on display and fade out on dismissal during normal step-advance (Next/Back) rather than appearing/disappearing instantly. This exit-fade requirement does not apply when the tooltip is torn down via Skip/Close per requirement 7.
14. The system shall detect the user's `prefers-reduced-motion` OS/browser setting and, when set to reduce, shall skip enter/exit transitions and modal/tooltip fades, showing/hiding content with no animation instead.
15. While the tour (or its completion modal) is open, Tab/Shift+Tab shall cycle focus only among the tour's own interactive controls (Next/Back/Skip, or the completion modal's controls) — focus shall never move to an element outside those controls until the tour or modal closes. Escape shall close/skip the tour from any focus state within this cycle.
16. The system shall not introduce any new runtime dependency for this feature other than a React-native tour engine (state kept in React component state, not an imperative library); modal and tooltip enter/exit animation shall be implemented as plain CSS transitions authored in Tailwind, not via an animation library. This constraint governs state ownership and animation mechanism only — it does not forbid the tour engine's own internal DOM reads used for positioning/spotlighting anchored elements (e.g., `getBoundingClientRect` calls for tooltip placement), which are distinct from a separate imperative DOM-manipulation/animation-orchestration library.
17. The system shall reuse the step content already expressed in `SELLER_WORKFLOW_STEPS` (the shared extraction of `app/playbook/page.tsx`'s `STEPS` array) as the single source of truth for tour step copy. Tour step copy (`CLOTHING_TOUR_STEPS` / `BOOK_TOUR_STEPS`) may be a lightly-adapted or paraphrased derivation of the corresponding `SELLER_WORKFLOW_STEPS` entry rather than a verbatim substring, provided it is demonstrably derived from that shared source and not independently authored as a duplicate array. Each tour step shall reference or map to a specific `SELLER_WORKFLOW_STEPS` index, and this mapping shall be verifiable via an automated test or a documented step-index mapping table.
18. The `/playbook` static page shall continue to function unchanged as a standalone read-only reference after this feature ships (existing behavior and its companion test in `app/playbook/__tests__/page.test.tsx` shall not regress).
19. The system shall automatically close the tour, without error, if the user switches item category (Book/Clothing) while the tour is open, since the tour's anchors belong to the form being replaced.
20. If the browser has no `localStorage` available (disabled, quota exceeded, private-mode restrictions), the system shall degrade gracefully: the tour remains fully usable, simply always presenting as not-yet-completed, without throwing an error or blocking the entry point.

## Non-functional requirements
- Tooltip/modal enter and exit transitions (per requirements 12-13) shall complete within 200ms, matching a "quick, not distracting" feel, implemented via Tailwind transition utility classes.
- The tour engine addition shall not add more than 50KB gzipped of client JS beyond the existing app bundle.
- All new interactive elements (tooltip, modal, controls) shall be keyboard operable with visible focus states, and shall respect `prefers-reduced-motion` (see requirement 14).
- No new external network calls or third-party services are introduced (project is local-first, no external services per README).
- Existing coverage thresholds (85/80/85/85 per project QA bar) shall be maintained or exceeded for all new/modified files.

## Constraints
- Must integrate with the existing add-item components `components/AddClothingForm.tsx` and `components/AddBookForm.tsx` (or a shared parent/page) without altering their existing field behavior, validation, or submission logic.
- Must reuse content already present in `app/playbook/page.tsx` and/or `docs/clothing-resale-research.md` Section 7 rather than duplicating it a third time.
- Styling must use Tailwind CSS 4 (already in use); no CSS-in-JS or new styling library.
- Framework constraints: Next.js 15 App Router, React 19, TypeScript — feature must work within existing client/server component boundaries (tour interactivity requires client components; the add-item forms' server/client split must not be broken).
- The only new runtime dependency permitted is a React-state-based tour engine (MIT-licensed); no AGPL-licensed libraries (e.g., Shepherd.js, Intro.js's free tier) and no framer-motion or other animation library.
- Must not modify or regress the existing `/playbook` page's current behavior or its test at `app/playbook/__tests__/page.test.tsx`.
- Must work with the project's existing dark mode (class-based) — tooltip/modal styling must support both light and dark themes.
- Test infrastructure constraint: any new tests must use `BOOKSELLER_DB_PATH`/`BOOKSELLER_PHOTOS_PATH` scratch overrides; must never point at the real `data/inventory.db`.
- Must pass the project's full QA bar: Vitest unit tests, Playwright E2E, Stryker mutation testing, `tsc --noEmit`, ESLint strict, and the "fallow" dead-code/complexity analyzer.

## Out of scope
- Animating or adding transitions to any existing UI outside the tour/tooltip/modal surfaces introduced by this feature (e.g., no general page-transition or route-change animation).
- A tour/tutorial for any workflow other than the add-item / presale prep flow (e.g., no tour for CSV import/export, dashboard filtering, or the sale-status state machine transitions).
- Personalization or adaptive tour content (e.g., skipping steps based on item category, showing different steps for books vs. clothing) beyond what's naturally implied by which form is being toured.
- Analytics/telemetry on tour engagement (step completion rates, drop-off tracking) — no such instrumentation exists in the project and none is introduced here.
- Server-side persistence of tour completion state across devices/sessions — persistence (FR 9) is local to the client only.
- Rewriting or restructuring the underlying content in `docs/clothing-resale-research.md` or the `STEPS` array beyond what's needed to make it reusable by both the static page and the tour.
- Multi-user/role-based tour variants — the app has a single operator persona, no distinct actor roles.
- Rewriting `AddClothingForm.tsx` / `AddBookForm.tsx` field structure or validation logic for reasons unrelated to adding tour anchor points.
- A dashboard-page entry point for the tour (may come in a future iteration).

## Acceptance criteria
1. Starting the tour from its entry point renders step 1's tooltip anchored to a real element in the add-item form; the underlying form remains fully visible and unmodified behind it.
2. Clicking "Next" through all steps reaches a final/completion state after no fewer than 5 and no more than 7 steps, each anchored to a distinct real form element.
3. Clicking "Skip"/"Close" at any step (first, middle, last) immediately and without any exit-transition animation removes all tour UI from the DOM and leaves the add-item form in the exact state (including any entered field values) it was in before the tour started. This immediate, non-animated teardown is by design and does not need to satisfy the exit-transition requirements of requirements 12-13.
4. Typing into a form field, then starting the tour, then clicking through 2+ steps, then skipping — the originally typed value is still present in the field afterward.
5. With OS/browser `prefers-reduced-motion: reduce` simulated, opening and closing the tour's modal/tooltip shows no animated transition (instant show/hide), verified via automated test asserting transition/animation CSS is suppressed or a reduced-motion code path is taken.
6. With `prefers-reduced-motion` not set (default), the modal/tooltip transitions on normal step-advance (Next/Back) and on the completion modal's own close button are visibly animated (non-zero, finite duration) on open and close; transitions triggered via Skip/Close remain instant per acceptance criterion 3.
7. Pressing Escape while the tour is open closes it in the same manner as clicking Skip (acceptance criterion 3 holds).
8. Completing or skipping the tour once for a given item category, then reloading the add-item page, does not auto-launch the tour again for that category; the manual entry point is still present and, when clicked, relaunches the tour from step 1. Completing/skipping the tour for one category does not affect the not-yet-seen state of the other category's tour.
9. `npm run build` (or equivalent) shows no new dependency added other than the single approved tour engine package; `package.json` contains no framer-motion, Shepherd.js, Intro.js, or Driver.js entries.
10. `app/playbook/page.tsx` renders identically to its pre-feature output and its existing test suite (`app/playbook/__tests__/page.test.tsx`) passes unmodified.
11. Each tour step's copy (in `CLOTHING_TOUR_STEPS` / `BOOK_TOUR_STEPS`) is verified, via an automated test or a documented step-index mapping table, to reference or derive from a specific index of `SELLER_WORKFLOW_STEPS` — a lightly-adapted/paraphrased derivation is acceptable, but an independently authored duplicate array is not.
12. `tsc --noEmit`, ESLint strict, Vitest (with coverage thresholds), Playwright E2E, and Stryker mutation testing all pass on the changed/added files.
13. Keyboard-only navigation (Tab/Shift+Tab/Enter/Escape, no mouse) can start the tour, advance through all steps, and skip/close it, with visible focus indication at each control.
14. Switching item category (Book/Clothing) while the tour is open closes the tour immediately without throwing an error, and the add-item form for the newly selected category is left in a normal, unmodified state.
