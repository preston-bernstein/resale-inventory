# Requirements: Marketplace Connector Tier

## Problem statement

`resale-inventory` tenants can now sign up, connect a marketplace platform, and give recorded consent to automate it — but no connector actually talks to a marketplace. The multi-tenant foundation (`docs/reseller-multi-tenant-foundation/requirements.md`) built the tenant/credential/consent/kill-switch machinery specifically so cross-listing automation could be layered on top of it safely; today that machinery gates nothing, because there is no automation to gate. This increment builds the two connector tiers — official-API (eBay, Etsy, Amazon) and browser-automation (Poshmark, Depop, Mercari, Vinted, Grailed) — behind one shared interface, so a tenant's inventory item can be created, updated, marked sold, or delisted on any of the 8 supported platforms, with every call passing through the existing consent/status gate first. No live third-party credentials exist yet for any platform; this increment wires real, testable connector code (eBay against its Sandbox, Etsy against its live-but-draft-only API, Amazon and the five browser-automation platforms in an inert/dry-run posture) so that supplying real credentials later is a configuration change, not a development project.

## Users / stakeholders

- **Tenant (seller)**: connects a marketplace account, triggers listing actions, and depends on the connector not getting that account banned or suspended.
- **Operator/owner (Preston)**: supplies platform-level developer-app credentials via environment variables; the only actor who can register real eBay/Etsy/Amazon developer apps or test marketplace accounts.
- **Existing `item_platforms` sync logic**: the current consumer this tier must not break — it already tracks which items are listed on which platforms and expects platform names drawn from `lib/constants.ts`'s `SUPPORTED_PLATFORMS`.
- **Kill-switch process (`lib/connections.ts`)**: `recordSuspensionSignal(tenantId, connectionId, reason, toStatus)` is the entry point connector code must call the moment a platform indicates suspension/ban; this tier is its first real caller.
- **Future connector maintainers**: anyone adding platform 9+, who should be able to implement one interface rather than learn two tiers' worth of ad hoc call patterns.

## Functional requirements

### Shared connector interface

1. The system shall define one TypeScript interface — `createListing`, `updateListing`, `markSold`, `delist`, `checkConnectionHealth` — implemented identically in signature (inputs/outputs/error shape) by every one of the 8 platform connectors, regardless of whether the connector is API-backed or browser-automation-backed.
2. The system shall expose a connector registry/factory (e.g. `getConnector(platform: SupportedPlatform)`) that returns the correct connector implementation for a given `platform_connections.platform` value, so caller code never branches on platform identity itself.
3. The system shall reject, at the type level or via a runtime guard covered by a test, any `platform` value passed to the connector factory that is not a member of `SUPPORTED_PLATFORMS`.
4. `createListing` shall accept item data already validated by existing `lib/money.ts`/`lib/transitions.ts`/category-detail conventions (integer cents, category-specific fields) and shall not perform its own parallel money or state validation.
5. `updateListing`, `markSold`, and `delist` shall each accept the platform's external listing identifier (as previously returned by `createListing` or stored in `item_platforms`) and the internal `tenant_id`/`connection_id`, and shall no-op with a typed "not found" result (not throw) when the platform reports the listing no longer exists.
6. `checkConnectionHealth` shall return a small typed status (e.g. `{ healthy: boolean, detail?: string }`) without mutating any listing state, and shall never throw a gating error (missing-consent or non-active-status) regardless of the connection's actual consent/status state — for a non-active or non-consented connection it shall return `{ healthy: false, detail: ... }` rather than throwing, so the call itself always completes. It shall not, however, act as a bypass of the gating requirements in the "Gating" section below for the other four methods.

### Gating (consent, connection status, credential isolation) — applies to all 8 connectors

7. The system shall, before any connector method other than `checkConnectionHealth` performs a network call or browser action, verify `hasValidConsent(tenantId, connectionId)` returns `true`; if `false`, the system shall throw/return a typed error identifying the missing-consent condition and shall make zero calls to the platform.
8. The system shall, before any connector method other than `checkConnectionHealth` performs a network call or browser action, verify the connection's `platform_connections.status` is `active`; if it is `suspended` or `revoked`, the system shall throw/return a typed error identifying the non-active status and shall make zero calls to the platform.
9. The system shall re-check consent and connection status per invocation, not once at connector construction time, so a status transition (e.g. a kill-switch suspension) that occurs between two `createListing` calls in the same batch/session blocks the second call even though the first succeeded.
10. The system shall retrieve platform-connection credentials exclusively via `getDecryptedCredential(tenantId, connectionId)` (or an equivalent function-level call requiring both parameters) — no connector may read `platform_connections.encrypted_credential` directly or cache a decrypted credential outside the scope of a single connector-method invocation.
11. The system shall never include a credential value (API token, OAuth token, session cookie, password) in a thrown error's message, a logged line, or any value returned to API-route callers — connector errors must carry platform error codes/messages only, scrubbed of request/response bodies that could contain the credential.
12. The system shall call `recordSuspensionSignal(tenantId, connectionId, reason, toStatus)` synchronously, from within the connector call that detected the signal, whenever a platform response unambiguously indicates account suspension, ban, or permanent API access revocation (e.g. eBay/Etsy/Amazon 401/403 responses carrying a suspension-specific error code; a browser-automation module detecting an account-restricted or suspended banner during a session) — `reason` shall be a short, non-secret string (platform + condition, e.g. `"ebay_403_account_suspended"`), never raw response bodies.
13. The system shall treat a ambiguous or transient error (network timeout, 500, rate-limit 429) as **not** a suspension signal — only errors the connector can positively classify as account-level suspension/ban shall trigger `recordSuspensionSignal`.
14. The system shall implement error/log/suspension-reason scrubbing via one shared utility function (not per-connector ad hoc logic) that strips any field a connector marks as credential-bearing before that data reaches a thrown error, a logged line, or the `reason` argument passed to `recordSuspensionSignal` — every connector shall route through this shared utility rather than reimplementing scrubbing itself.
15. The system shall validate `tenantId`, `connectionId`, and `itemId` values passed into any connector method as well-formed UUIDs before using them in any database query or Playwright navigation/selector construction, and all SQL access performed by this tier shall use parameterized queries — never string-interpolated SQL.

### API tier — shared

16. The system shall source each platform's app-level developer credentials (distinct from any tenant's personal marketplace login) from environment variables, with placeholder names documented in `.env.example` and `README.md`, and shall fail fast with a clear startup/first-call error (not a silent no-op) when a required app-level env var is unset and a tenant attempts to use that platform's connector. This fail-fast behavior applies identically and independently to all 3 API-tier connectors (eBay, Etsy, Amazon), each of which shall throw its own distinct typed configuration error (e.g. `EbayNotConfiguredError`, `EtsyNotConfiguredError`, `AmazonNotConfiguredError` per requirement 28), and each shall have equivalent test coverage — this is not an Amazon-only pattern.
17. The system shall perform each API-tier connector's OAuth token exchange and refresh using only the tenant's own stored credential (refresh token / auth code) retrieved via `getDecryptedCredential`, storing any newly issued token back through `rotateCredential(tenantId, connectionId, ...)` rather than holding it only in memory.
18. The system shall, when an OAuth token refresh attempt itself fails with a response indicating the refresh token or authorization has been revoked (not merely that the access token expired), classify that failure per the same suspension-classification rules as requirements 12-13 — calling `recordSuspensionSignal` only when the refresh-failure response is positively classified as revocation, and treating a generic or transient refresh failure (network error, 500, timeout) as not a suspension signal.

### eBay connector

19. The system shall implement the eBay connector against eBay's Sandbox environment (`api.sandbox.ebay.com`), using OAuth 2.0 with the `sell.inventory` scope, calling the Sell Inventory API's Create/Replace Inventory Item, create Offer, and publish Offer operations for `createListing`.
20. The system shall read the eBay Sandbox base URL and app-level Client ID/Client Secret from environment variables (e.g. `EBAY_SANDBOX_CLIENT_ID`, `EBAY_SANDBOX_CLIENT_SECRET`, `EBAY_ENV`), defaulting `EBAY_ENV` to `sandbox` so a misconfiguration cannot silently hit eBay production.
21. The system shall implement `updateListing` via the Sell Inventory API's Replace Inventory Item / update Offer operations, `markSold` by ending the Offer with a sold reason where the Sandbox supports it, and `delist` by withdrawing the Offer.
22. The system shall have integration tests that exercise `createListing`/`updateListing`/`markSold`/`delist`/`checkConnectionHealth` against the real eBay Sandbox (not a mock), gated behind the presence of sandbox app credentials in the test environment, and shall skip (not fail) those tests when sandbox credentials are absent so `npm test` remains runnable with none configured.

### Etsy connector

23. The system shall implement the Etsy connector against Etsy Open API v3, using OAuth 2.0 Authorization Code Grant with mandatory PKCE, sourcing the app-level API key and shared secret from environment variables (e.g. `ETSY_API_KEY`, `ETSY_SHARED_SECRET`).
24. The system shall create every listing via `createListing` in Etsy's native draft/inactive state (`state: "draft"`) and shall never transition a listing to Etsy's active state as part of any automated connector call — activating a real live Etsy listing requires a separate, explicitly-invoked action outside this connector's default `createListing`/`updateListing` behavior, and no such activation action is implemented in this increment. This draft-only behavior is a hardcoded default for this increment — no configuration flag, toggle, or connector option exists to change it; enabling live-state creation would require a future increment's explicit design and build.
25. The system shall document, in code comments and in this connector's README section, that Etsy has no sandbox environment, and that `createListing`/`updateListing` calls against a tenant's real Etsy account therefore create real (draft-state) listings on Etsy's live system the moment real tenant credentials are configured — draft state, not a sandbox boundary, is what prevents unintended public listings.
26. The system shall implement `markSold` and `delist` against Etsy Open API v3's listing-state-change endpoints, scoped to listings the connector itself created (tracked via `item_platforms`' external-id linkage), and both shall be no-ops (typed "not found" result) against a listing already in a terminal Etsy state. Both methods shall operate correctly against listings still in Etsy's `draft` state (Etsy allows deleting/updating a draft listing's state), so `markSold`/`delist` are fully testable this increment without ever activating a listing to Etsy's live/active state.

### Amazon SP-API connector (wired, inert)

27. The system shall implement an Amazon SP-API client module and LWA (Login with Amazon) token-exchange flow implementing the same 5-method interface, reading app-level LWA Client ID/Client Secret and SP-API role/refresh-token configuration from environment variables (e.g. `AMAZON_LWA_CLIENT_ID`, `AMAZON_LWA_CLIENT_SECRET`, `AMAZON_SP_API_REFRESH_TOKEN`).
28. The system shall have the Amazon connector's every method (`createListing`, `updateListing`, `markSold`, `delist`, `checkConnectionHealth`) throw a distinct, typed `AmazonNotConfiguredError` (not a generic error, not a silent success) whenever the required environment variables are unset, so the connector's inert state is explicit and machine-detectable rather than a bug waiting to be discovered — following the same fail-fast-with-typed-error pattern applied to eBay and Etsy per requirement 16.
29. The system shall have unit tests for the Amazon connector's request-building, LWA token-exchange logic, and error-mapping logic that run entirely against mocked HTTP responses (no live SP-API calls, since no Professional Selling Plan or Developer Profile exists) and shall document in the connector's code comments and README that live verification is blocked on the account owner completing SP-API developer registration.

### Browser-automation tier — shared

30. The system shall implement each of the 5 browser-automation connectors (Poshmark, Depop, Mercari, Vinted, Grailed) as a Playwright-driven module implementing the same 5-method interface, using `playwright` (added as a production dependency, distinct from the existing `@playwright/test` devDependency used for this repo's own e2e suite).
31. The system shall default every browser-automation connector to dry-run mode (constructs and logs the action it would take; makes no real browser navigation or submission) whenever the tenant's stored credential record for that platform+tenant connection is absent, OR its stored value is a literal sentinel marker — a reserved string/flag persisted alongside the encrypted credential (e.g. a `credential_status: 'placeholder'` marker) — explicitly indicating that no real credential has been configured yet. Dry-run determination shall be this explicit marker check, never an inferred heuristic on the credential value's shape or content. Real browser automation shall only run once a tenant has supplied a real credential and the sentinel marker has been cleared.
32. The system shall persist each platform's authenticated browser session (cookies/local storage) per tenant+connection via the existing encrypted credential table (as part of the stored credential payload, still encrypted at rest, never in plaintext on disk outside that table), and shall reuse a persisted session on subsequent calls rather than performing a fresh login for every action.
33. The system shall perform a fresh login only when no valid persisted session exists or the persisted session is rejected by the platform (e.g. redirected to a login page). One login "attempt" means one full navigate-to-login-and-submit-credentials cycle; a fresh login shall never be attempted more than once per connector-method call, and if that single cycle fails (rejected credentials, unexpected page state, timeout) the method shall surface a typed error rather than retrying the login automatically.
34. For the 5 browser-automation connectors specifically, `checkConnectionHealth` shall validate an existing persisted session read-only and shall never attempt a fresh login as a side effect of a health check — fresh login (requirement 33) is reserved exclusively for the mutating methods' (`createListing`/`updateListing`/`markSold`/`delist`) session-establishing flow. This prevents health-check polling from becoming an unthrottled, undetected source of bot-detectable login attempts.
35. The system shall apply a conservative-pacing posture on all 5 browser-automation platforms: no connector action may run at a fixed retry-immediately loop on failure, and every platform module shall log each rate/cooldown decision it makes, at minimum recording platform, connection id, action type, and the pacing/cooldown decision made (allowed/blocked, plus reason) — not full request/response payloads. Each browser-automation connector's methods shall navigate only to the specific listing/item page(s) required for the requested action, and shall never enumerate or fetch the tenant's full closet/catalog as a side effect of `createListing`/`updateListing`/`markSold`/`delist`/`checkConnectionHealth` — no connector shall poll or scrape more of a tenant's account than needed to manage that tenant's own listings.
36. The system shall have no test accounts for any of the 5 browser-automation platforms in this increment; each connector's Playwright selectors/flows shall be built and reviewed against that platform's real current UI (captured via the same non-mutating browser-inspection approach used to build the module, not against a mock DOM fixture) but shall be exercised in automated tests only in dry-run mode.
37. Browser-automation connectors shall construct Playwright interactions using value-based APIs (e.g. `fill`, `check`, role- or test-id-based locators) rather than interpolating tenant-supplied listing text (title, description, or any other free-text field) directly into dynamically-constructed selector strings.
38. Each browser-automation connector call shall use a browser context/cookie-jar instantiated fresh and scoped to the specific `tenantId`+`connectionId` pair being acted on — session state for one tenant+connection shall never be reused or shared across a different tenant or connection.

### Poshmark rate-limit and ban-risk mitigation (hard requirement)

39. The system shall track, per tenant+Poshmark connection, each delist event keyed by item+connection, and shall refuse (typed cooldown error, not a silent skip) to relist an item whose most recent delist event for that connection occurred fewer than 60 days ago, consistent with Poshmark's May-2025 policy penalizing excessive delist/relist within a 60-day window.
40. The system shall track, per tenant+Poshmark connection, a rolling 24-hour share-action count and shall stop issuing further share actions once that count reaches a configurable cap defaulting to 3500 (a conservative margin below Poshmark's ~4000 shares/day "share jail" threshold), resuming only after the rolling window clears.
41. The system shall provide a Poshmark-specific `sharePoshmarkListing(tenantId, connectionId, itemId)` action, as an addition outside the shared 5-method interface (requirement 1), gated by the same consent/status/credential rules that apply to the 5 shared methods (requirements 7-13), and subject to the rolling 24-hour share-count cap tracked by requirement 40.
42. The system shall persist the delist/relist and share-count tracking state used by requirements 39-40 in a durable store (SQLite table, following the existing additive-migration/UUIDv4 convention) — not in-process memory — so the cooldown survives a process restart.

### Depop, Mercari, Vinted, Grailed pacing posture

43. The system shall apply the same session-reuse-over-repeated-logins and no-more-scraping-than-needed posture (requirement 35) to Depop, Mercari, Vinted, and Grailed, and shall apply a per-platform configurable action-rate cap (default: no more than 1 create/update/markSold/delist action per tenant+connection per 10 seconds) as a conservative default in the absence of each platform's own documented thresholds.
44. The system shall document, in each of the 4 connectors' code comments, that the specific numeric thresholds used are conservative defaults chosen in the absence of a published platform policy (unlike Poshmark's documented 60-day/4000-share thresholds), and shall make each threshold a single named constant in `lib/constants.ts` (following this repo's existing constants-consolidation convention) so it can be tightened or loosened later without a code search.

### Integration with existing `item_platforms` and kill-switch

45. The system shall, on a successful `createListing`, write or update the corresponding `item_platforms` row (linking `item_id` to `platform`) using the platform's returned external listing identifier, following the existing `UNIQUE(item_id, platform)` constraint and `listed_at` convention — this tier shall not introduce a parallel listing-tracking table that duplicates `item_platforms`' purpose.
46. The system shall expose the platform's external listing identifier returned by `createListing` in a form the existing `item_platforms` write path can consume without modifying `item_platforms`' schema.
47. The system shall be independently exercisable per platform: a failure or misconfiguration in one platform's connector (e.g. Amazon's inert state, or a missing eBay sandbox credential) shall not prevent any other platform's connector from being constructed, gated, or invoked.

## Non-functional requirements

- `checkConnectionHealth` for API-tier connectors (eBay, Etsy) must complete within 3s at p95 against their respective live/sandbox endpoints, excluding cases where the upstream platform itself is slow (not a connector-side budget the code can guarantee, but the connector must not add its own retry/backoff delay inside this call).
- Browser-automation connector actions (`createListing`, `updateListing`, `markSold`, `delist`) must not be retried automatically more than once within a single method invocation — automatic retry loops are a documented Poshmark ban-risk vector and are disallowed platform-wide in this tier, not just for Poshmark.
- Automated tests (`npm test`) shall never launch a real Playwright browser instance — dry-run mode and/or module-level mocking of the `playwright` import shall guarantee this deterministically, not merely by relying on credentials being absent from the test environment.
- No credential (API key/secret, OAuth token, session cookie) may appear in plaintext in a log line, thrown error, or Playwright trace/video artifact — Playwright tracing/video capture, if enabled for debugging, must be scoped to dry-run sessions only or must have credential-bearing network requests explicitly excluded from the trace.
- All new tables (Poshmark rate/cooldown tracking, per-platform pacing state) must be added via additive `CREATE TABLE IF NOT EXISTS` migrations under `data/migrations/`, gated by an incremented `PRAGMA user_version`, consistent with `005_tenants.sql` through `008_consent_capture.sql`.
- New enum-like columns must use inline SQLite `CHECK` constraints; no `ALTER ... CHECK` — same convention as `platform_connections.status`.
- Connector code must not introduce floating-point money handling; any price data a connector sends to or receives from a platform API must convert cents via `lib/money.ts` at the connector boundary only.
- The eBay Sandbox test suite must be safe to run repeatedly without manual sandbox cleanup between runs (idempotent test data, e.g. UUID-suffixed SKUs) since Sandbox mimics production and accumulates state.

## Constraints

- Must consume, not rebuild, the multi-tenant foundation's tables and functions: `platform_connections` (`data/migrations/007_platform_connections.sql`), `tenant_consents`/`disclosure_versions` (`008_consent_capture.sql`), `hasValidConsent`/`recordConsent`/`revokeConsent` (`lib/consent.ts`), `createConnection`/`getConnection`/`getDecryptedCredential`/`rotateCredential`/`recordSuspensionSignal`/`reactivateConnection` (`lib/connections.ts`), and `encryptCredential`/`decryptCredential` (`lib/credentialCrypto.ts`).
- Must use the existing `SUPPORTED_PLATFORMS` allowlist in `lib/constants.ts` as the sole source of valid platform identifiers — do not introduce a second platform-name enum.
- Must follow the existing migration file convention (numbered files under `data/migrations/`, `PRAGMA user_version` gating in `lib/db.ts`'s `VERSIONED_MIGRATIONS` array) for any new tables this tier requires (Poshmark cooldown/share tracking).
- Must preserve all invariants in `.claude/skills/resale-inventory-architecture-contract/SKILL.md`: integer-cents money via `lib/money.ts`, UUIDv4 PKs with the existing length/version CHECK pattern, inline CHECK constraints (no ALTER CHECK), the `status` state machine in `lib/transitions.ts` untouched, `category` immutability, and "sacred DB" rules (never delete/recreate `data/inventory.db`).
- Must not break `BOOKSELLER_DB_PATH` scratch-DB test isolation — connector tests (including the real eBay Sandbox integration tests) must run against the scratch DB for all `platform_connections`/`item_platforms`/tenant data, even though the eBay Sandbox itself is a real external environment.
- Must not weaken any consent/status gate from the multi-tenant foundation (requirements 14-28 there) — this tier is the enforcement point that gate exists for.
- Platform-level app credentials (eBay dev keys, Etsy API key/secret, Amazon LWA client ID/secret) come from environment variables only; this increment must not invent, fetch, or hardcode real values — placeholders only, documented in `.env.example`/`README.md`.
- Must add `playwright` as a production dependency (`package.json` `dependencies`, not just the existing `@playwright/test` devDependency) to run browser automation outside the test runner; this is a new runtime dependency and must be called out in `README.md`, consistent with the app's existing "no external services required to run it" framing being about services, not libraries.
- No live third-party credentials exist for any of the 8 platforms; Amazon and all 5 browser-automation platforms must ship in an inert/dry-run-by-default state that requires no live credential to build, test, or deploy successfully.
- `.env`/`.env.local` (or equivalent) must already be covered by the repo's `.gitignore` before `.env.example` is added — verify this as part of the work, do not assume it.

## Out of scope

- Obtaining, registering, or configuring any real eBay production app, Etsy Seller App, Amazon Developer Profile/Professional Selling Plan, or any of the 5 browser-automation platforms' real tenant login credentials — these remain the account owner's task outside this build. This refers to production/live marketplace accounts and app registrations; it does not cover eBay's free self-serve Sandbox app registration, which is a separate, lighter-weight task the operator (Preston) must still complete outside this build to exercise the Sandbox-gated tests required by requirement 22.
- Activating (publishing to Etsy's live/active state) any Etsy listing this connector creates — the connector defaults to draft-only and no activation path is built this increment.
- Any live, credentialed exercise of the Amazon connector — it is wired and unit-tested against mocks only.
- Any live, credentialed exercise of the 5 browser-automation connectors against real marketplace accounts — they run in dry-run mode by default and are validated against real UI structure, not real live sessions, in this increment.
- A UI/dashboard surface for tenants to trigger connector actions or view listing sync status — this increment builds the connector layer only; wiring it into `app/api/items` routes or a UI is a follow-on increment.
- Cross-platform price/inventory-sync orchestration logic (e.g. auto-delisting from all platforms when one platform reports Sold) — this increment provides the primitives (`markSold`, `delist`) that such orchestration would call, but does not build the orchestration itself.
- CAPTCHA-solving, proxy rotation, or anti-bot-detection evasion infrastructure for the browser-automation tier — conservative pacing is the mitigation strategy here, not evasion tooling.
- Etsy's or eBay's non-listing APIs (order management, shipping label purchase, returns) — only the 5-method connector surface (create/update/markSold/delist/health) is in scope.
- A generalized platform-agnostic scraping/analytics feature beyond what each connector needs to manage the tenant's own listings.

## Acceptance criteria

1. Calling any of `createListing`/`updateListing`/`markSold`/`delist` for a tenant+connection with no valid consent record throws/returns the missing-consent error and results in zero network/browser calls to the platform — verified by a test asserting an HTTP mock/Playwright launch spy was never invoked.
2. Calling any of `createListing`/`updateListing`/`markSold`/`delist` for a tenant+connection whose `platform_connections.status` is `suspended` or `revoked` throws/returns the non-active-status error and results in zero network/browser calls — verified the same way as (1), for both statuses.
3. A connection that is `active` with valid consent when `createListing` starts, but is suspended by a concurrent `recordSuspensionSignal` call before a second action (`updateListing`) in the same test runs, has the second action blocked — verified by a test that suspends mid-sequence and asserts the second call is rejected.
4. No connector method, under any error path exercised by its test suite, includes a seeded credential/token/session-cookie string in a thrown error message or console log — verified by a test that seeds a known secret, triggers each method's error paths, and asserts the secret string does not appear in captured output.
5. The eBay connector's `createListing`/`updateListing`/`markSold`/`delist`/`checkConnectionHealth` each successfully round-trip against the real eBay Sandbox when sandbox credentials are present in the test environment, and each of those tests is skipped (not failed, not erroring) when sandbox credentials are absent.
6. The Etsy connector's `createListing` creates a listing in Etsy's `draft` state and no code path in `createListing` or `updateListing` sets Etsy's listing state to active — verified by a test asserting the request payload sent to Etsy always carries `state: "draft"` (or equivalent) and never `"active"`.
7. Every Amazon connector method throws `AmazonNotConfiguredError` when the required LWA/SP-API environment variables are unset, verified by a test running with those variables unset that asserts each of the 5 methods throws this specific error type.
8. Each of the 5 browser-automation connectors runs in dry-run mode (no real Playwright browser launch/navigation) when the tenant's stored credential is absent or marked as a dry-run credential, verified by a test asserting no browser context was created.
9. A simulated Poshmark relist within 60 days of that same item's delist on the same tenant+connection is rejected with a typed cooldown error, verified by a test seeding a delist event 30 days prior and asserting the relist attempt fails; a relist attempted at 61+ days succeeds.
10. A simulated sequence of Poshmark share actions that reaches the configured rolling-24-hour cap has the next share action rejected with a typed cooldown error, verified by a test that drives the counter to the cap and asserts the next call is rejected while a call after the window resets succeeds.
11. A simulated sequence of actions for one of the Depop/Mercari/Vinted/Grailed connectors that exceeds the configured per-platform rate cap (requirement 43) has the next action rejected with a typed rate-limit error, verified by a test that drives the limiter to its cap and asserts the next call is rejected while a call after the window resets succeeds.
12. A successful `createListing` call for any of the 8 platforms results in exactly one `item_platforms` row for that `item_id`+`platform` pair, with the platform's external listing id captured, verified by a test asserting the row exists post-call and a second `createListing` call for the same item+platform does not create a duplicate row (respects `UNIQUE(item_id, platform)`).
13. A platform response classified by a connector as an account-suspension signal results in exactly one call to `recordSuspensionSignal` with a `reason` string containing no credential material, verified by a test that mocks a suspension-shaped response and asserts the call and its arguments.
14. A platform response that is a generic 500/timeout/429 does not trigger `recordSuspensionSignal`, verified by a test asserting the mock is not called for these response types.
15. Running `npm test` with no real eBay/Etsy/Amazon/browser-automation credentials configured in the environment passes in full (Sandbox and dry-run/mocked paths only), demonstrating the "wired but inert until credentialed" requirement holds without any live secret.
16. `.env.example` documents a placeholder entry for every platform-level app credential referenced by any connector (eBay, Etsy, Amazon), and `README.md` states, for each of the 8 platforms, its current operability tier (real-Sandbox-tested / live-draft-only / inert-until-credentialed / dry-run-until-credentialed).
