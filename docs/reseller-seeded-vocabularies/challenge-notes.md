## Changes made
- **Fixed a real optional-vs-required-field bug before it shipped**: `validateColorInput`/`validateMaterialInput`/`validateDepartmentInput` were specified as literal copies of `validateBrandInput`, which requires non-empty input — but color/material/department are optional fields. A literal copy would have rejected every clothing submission that omits one of them (the common case). Now explicit: undefined/empty is valid, no resolve call made.
- **Resolved a self-contradicting instruction about `validateGenderDepartment`**: the original plan said this existing, multiply-referenced function (also used by PATCH `/api/items/[id]` and `POST /api/import`) "gains" a length cap — but the integration-points list separately implied a brand-new, separate function instead. Now explicit: the existing function is untouched; a new `validateDepartmentInput` in `lib/departments.ts` is used only by the new POST-create path. PATCH/import keep their existing behavior, documented as a pre-existing, out-of-scope gap rather than something silently changed.
- **Removed unnecessary migration risk**: the new tables' `id` column no longer copies `clothing_brands`' UUIDv4-version-nibble CHECK constraint (which required a fragile `randomblob`/`hex`/`substr` SQL expression the plan itself flagged as the top risk area). Nothing functionally needs a v4-shaped id for these lookup tables — switched to a plain 32-hex-char random id, eliminating that risk entirely.
- **Caught a real SQL-escaping gap before it could break the migration**: the seed value lists contain apostrophes ("Levi's", "Men's", "Women's", "Kids'") but the plan's only worked SQL example (colors) had none — a single un-escaped apostrophe in the real migration file would roll back the whole transaction. Plan now spells out the escaped literals explicitly.
- **Consolidated three near-duplicate resolver files into one shared factory** (`lib/vocabResolver.ts`) instead of hand-copying `lib/brands.ts`'s resolve/validate/select trio three times — avoids a near-certain fallow duplication flag later (this repo runs `fallow audit` with `minOccurrences: 2` in CI) and avoids "any bugfix needs 3-4 coordinated edits" risk.
- **Fixed steps.md's granularity and a broken idempotency test**: split the overloaded Step 1 into 1a/1b/1c (DDL, new-table seeding, brand backfill — different risk profiles), merged three redundant lib-file steps into one and three redundant endpoint steps into one, and fixed a test that would have failed on its own first assertion (re-running the whole migration `.sql` file errors on `CREATE TABLE` before ever reaching the `INSERT OR IGNORE` it was meant to verify).
- **Added a missing index and a missing length cap** to the new tables' schema (`clothing_details(tenant_id)` was unindexed despite this feature tripling query volume against it; the new tables' `canonical_name` had no upper-bound CHECK matching the app-layer 255-char rule).

## Critiques rejected
- **Design Devil's Advocate's suggestion to revert the `gender_department` field to a closed `<select>` enum** (rather than the same open VocabCombobox + add-new pattern as color/material) — a legitimate design concern (typo-driven duplicate departments on a field with a genuinely small, known taxonomy), but rejected because it contradicts the user's explicit product direction: uniform preseeded-vocabulary + add-new-choice behavior across every clothing form field, stated directly by the user this session. Documented in plan.md as a conscious, accepted tradeoff rather than a silent gap.
- **Design Devil's Advocate's "seed-on-first-read" alternative** to the dual migration-backfill + signup-time-seeding architecture — a real simplification (single source of truth, no frozen-SQL-vs-live-JS-array duplication risk) but rejected as too large a change for this pass: the requirements doc's FR4-8 and their acceptance criteria (AC2, AC3, AC5) explicitly specify the two-mechanism design as a testable requirement, not just an implementation detail up for revision.
- **Security auditor's XSS-escaping concern for the "Add {typed}" combobox option** — not applicable; this is a React/JSX codebase where all text content is escaped by default (no `dangerouslySetInnerHTML` anywhere in this pattern), matching how `BrandCombobox`'s already-shipped identical display already works safely.
- **Data Model Critic's brand-seeding case-collision edge case** (a tenant with an already-resolved lowercase brand silently keeping that casing after the backfill's `INSERT OR IGNORE`) — accepted as a real, documented risk in plan.md, but not fixed with reconciliation logic, since real production usage of the brand field is near-zero (PR #12 just shipped).
- **Various pre-existing gaps flagged as blockers but already present in the shipped brand feature** (no Unicode/control-character policy on canonical_name, verbose validation error responses, no per-tenant creation rate limit) — these match `clothing_brands`' existing, already-accepted behavior exactly; fixing them here would be scope creep beyond what this feature changes.
