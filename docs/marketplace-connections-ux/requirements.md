# Requirements: Marketplace Connections UX

## Problem statement

`resale-inventory` tenants can already authenticate, and the backend can create, consent-gate, kill-switch, and (per-platform) drive connections to all 8 supported marketplaces (`eBay`, `Etsy`, `Amazon`, `Poshmark`, `Depop`, `Mercari`, `Vinted`, `Grailed`) — but no page in `app/` exposes any of it. A tenant today has no way to connect a marketplace, see its risk disclosure, grant or revoke consent, see whether a connection is healthy or killed, or confirm a connection actually attached to the right account. This is the last gap between a shipped backend and a usable product: without this UX, none of the multi-tenant foundation or connector tier is reachable by an actual seller. It matters now because the two connector tiers carry genuinely different risk (OAuth API access vs. credential/session-based browser automation that can get a seller's account banned), and presenting them identically would either under-warn sellers connecting a credential-tier platform or over-warn sellers connecting a low-risk OAuth platform — both erode trust at the exact moment a seller is deciding whether to hand over marketplace access.

## Users / stakeholders

- **Tenant (seller)**: the only user of this page. Connects marketplace accounts, reads and acts on risk disclosures, monitors connection health, reconnects or reactivates when a connection breaks.
- **Existing consent/kill-switch backend** (`lib/consent.ts`, `lib/connections.ts`, `data/migrations/007_platform_connections.sql`, `008_consent_capture.sql`): this UX is a pure consumer — it must not require schema or route changes to `app/api/connections/**` or `app/api/disclosures/current`.
- **Existing connector tier** (`lib/connectors/*`): its `checkConnectionHealth` and the `item_platforms` table it writes to are the only two connector-layer facts this UX may read; it must not call `createListing`/`updateListing`/`markSold`/`delist` directly (no listing-trigger API route exists yet — see Out of Scope).
- **Operator/owner (Preston)**: cares that this page cannot be used to bypass consent gating or leak credential material client-side, since it is the first UI surface with access to connection state.

## Functional requirements

### Empty state

1. The system shall render a `/connections` page that, when `GET /api/connections` returns an empty array for the resolved tenant, shows a single empty state with exactly one primary call-to-action labeled to connect the tenant's first marketplace — no per-platform cards, status list, or other primary action shall be visually competing with it in this state.
2. The system shall distinguish this "zero connections ever" empty state from any post-connection state (one or more rows returned) — once at least one connection exists, the empty-state CTA shall not reappear, matching the existing `ItemCardGrid` convention of separate empty-state messaging for "nothing exists yet" vs. other zero-result states.
3. The empty-state CTA shall navigate to (or expand into) the per-platform connect-card view (requirement 4), not directly into any single platform's consent screen — the tenant must still choose a platform.

### Per-platform connect cards, split by trust tier

4. The system shall render exactly 8 connect cards, one per member of `SUPPORTED_PLATFORMS` (`lib/constants.ts`), grouped into two visually distinct sections: an OAuth-tier section (`eBay`, `Etsy`, `Amazon`) and a credential/session-tier section (`Poshmark`, `Depop`, `Mercari`, `Vinted`, `Grailed`) — the grouping shall be static (derived from a fixed tier map, not a runtime API call), since the OAuth-vs-credential split is a build-time fact, not per-tenant state.
5. Each connect card shall display, at minimum: platform name, its trust tier (OAuth or credential/session), and its current per-tenant connection state if one exists (not connected / active / suspended / revoked), sourced from `GET /api/connections` filtered client-side by `platform`.
6. Each connect card's action shall be disabled (not hidden) when a connection for that platform already has `status` `active` or `suspended` for this tenant — `POST /api/connections` returns `409 connection_exists` for exactly these two states, so the UI shall not allow an action that the API will reject; the card shall instead route to the status view (requirements 15-21) for that connection.
7. Each connect card's action shall remain enabled when no connection exists for that platform, or when the existing connection's `status` is `revoked` — both cases succeed against `POST /api/connections` (plain create, or the delete-then-recreate reconnect path), per that route's documented three-case contract.
8. Selecting a connect card's action shall route to that platform's risk-disclosure/consent screen (requirement 9) before any `POST /api/connections` call is made — no connect card action shall call `POST /api/connections` directly.

### Risk-disclosure / consent screen

9. Before submitting a credential for any platform (OAuth or credential-tier — the backend's `hasValidConsent` gate is uniform across all 8 platforms per the connector tier's gating requirements), the system shall fetch `GET /api/disclosures/current` and render its `content` and `version` to the tenant, and shall not proceed to credential submission until the tenant performs an explicit affirmative action on that specific platform's screen — no pre-checked or default-accepted control.
10. The tenant's affirmative action on the consent screen (requirement 9) is a client-side-only gesture — it makes no network call. The full sequence is: disclosure shown → tenant affirms on-screen (client-only, no network call) → credential fields shown → on submit, the system calls `POST /api/connections` (create) followed by `POST /api/connections/:id/consent` (record), using the `disclosure_version` captured at affirm-time, not re-fetched at submit time (requirement 13 specifies this call in full). The "explicit affirmative action" required by requirement 9 is this client-side gesture; it is distinct from — and happens well before — the later `POST /api/connections/:id/consent` API call that persists it.
11. The consent screen shall present the shared disclosure `content` alongside platform-specific risk framing (e.g., ban/suspension risk language distinguishing OAuth from credential/session tiers) — since `disclosure_versions.content` is one global, non-platform-scoped text field (per `data/migrations/008_consent_capture.sql`), the per-platform framing is UI copy layered over the shared text, not separate stored content; the same `version` number is what gets recorded regardless of platform.
12. The system shall not allow the tenant to grant consent for more than one platform in a single action — each platform connection's consent is submitted via its own `POST /api/connections/:id/consent` call with that connection's own `id`, even when multiple platforms reference the same current `disclosure_version` — no "accept all" control shall exist.
13. Submitting consent shall call `POST /api/connections/:id/consent` with `{ disclosure_version }` set to the value captured once, at the moment the tenant affirmed the consent screen (requirement 10) — carried through client-side state from affirm-time to this call, not re-fetched from `GET /api/disclosures/current` immediately before submission — after the connection row itself has been created via `POST /api/connections` (consent is recorded against a `connection_id`, which does not exist until the connection is created). The create-connection call and the record-consent call shall both occur before the connection is treated as usable, and the UI shall surface `422 invalid_disclosure_version` / `422 stale_disclosure_version` as a retry-the-consent-step error, not a generic failure — server-side staleness, if the disclosure version changed between affirm-time and submission, is exactly what this stale-version check catches.
14. The system shall not present the credential input fields (platform-specific fields, e.g. from a per-platform variant of `CredentialFields`) until after the consent screen has been affirmatively completed for that platform connection attempt.

### Connection status view

15. The system shall render each existing connection's `status` (`active` / `suspended` / `revoked`) using a 4-color severity convention: green = `active`, yellow = `suspended`, red = `revoked`, and blue reserved for informational states outside the `connection_status` enum itself (e.g., a connection with `active` status but no valid current-version consent, or a credential-tier connection running in dry-run mode per its `credential_status` sentinel) — blue shall never be used to represent any of the three enum values directly, since the enum has only three members and the convention calls for four colors.
16. For each connection returned by `GET /api/connections`, the system shall additionally fetch `GET /api/connections/:id/consent` to obtain that connection's `has_valid_consent` value, so the two distinct signals required by the next requirement can be rendered for every connection in the full list — not only for a single connection viewed in isolation.
17. For each connection, the status view shall separately surface (a) `connection_status` (via the color convention above) and (b) current consent validity (via `GET /api/connections/:id/consent`'s `has_valid_consent` boolean) as two distinct signals, since a connection can be `active` with stale/missing consent (blocked from automation per the backend gate) — these two facts shall not be collapsed into one indicator.
18. A connection with `status` `active` but no valid current consent (per the previous requirement's signal) — e.g., because `POST /api/connections` succeeded and the connection row exists, but the subsequent `POST /api/connections/:id/consent` call never completed (the tenant closed the tab, or a non-422 error occurred) — shall render a visible action allowing the tenant to resume the consent step for that specific existing connection, routing to the consent screen (requirement 9) for that connection's `id`, rather than leaving the tenant stuck or forcing them into an unrelated flow.
19. A connection with `status` `suspended` shall render a visible reactivate action that calls `POST /api/connections/:id/reactivate`, and shall surface `409 not_suspended` (returned when the connection is already active or is revoked) as an error state rather than a silent no-op.
20. A connection with `status` `revoked` shall not render a reactivate action (the backend does not support reactivating a revoked connection — only `suspended → active` is modeled) and shall instead route the tenant directly into that platform's risk-disclosure/consent screen (requirement 9), skipping the connect-card view — since the tenant already chose that platform by triggering this reconnect action, forcing an extra click through the connect card they already selected is unnecessary friction (requirement 7's reconnect path).
21. The status view shall display each platform's static operability tier (real-Sandbox-tested / live-draft-only / inert-until-credentialed / dry-run-until-credentialed, per `README.md`'s Marketplace Connectors table) as informational (blue) text distinct from `connection_status`, since these are two independent axes — a connection can be `active` (backend-status) while its platform is still `inert-until-credentialed` (operability tier) if no app-level environment credential has been configured.

### Masked-identifier confirmation

22. Immediately after a successful `POST /api/connections` (`201`), the system shall display a masked confirmation of the platform identifier the tenant entered in that submission's credential fields (e.g., `connected as @{handle}` with all but a fixed leading/trailing portion of the value obscured) — sourced from the client-side form state the tenant just submitted, not from any API response, since `platform_connections` credential values are never returned by any route (`ConnectionMetadata` carries no credential or identifier field, per `lib/connections.ts`'s `toMetadata`).
23. The masking rule referenced by requirements 22 and 26 (and tested by acceptance criteria 12 and 15) shall produce a non-degenerate mask for an identifier of any length ≥1 character — e.g., a fixed-width mask for identifiers too short to show distinct leading and trailing characters — so no submitted identifier value can appear unmasked or with an ambiguous-length mask in the rendered confirmation, regardless of the identifier's length.
24. The credential-entry step (requirement 14) shall require a non-secret platform-identifier field (e.g. username/handle) as one key of the submitted `credential` object, in addition to whatever secret fields that platform's tier requires — this is compatible with the existing validation, since `assertValidCredentialShape` only requires a non-null, non-array object and does not constrain which keys it contains.
25. The credential-entry fields required by the previous requirement — the non-secret platform-identifier field and each platform's required secret fields — shall be validated as non-empty on the client before the submit action is enabled; this is a basic required-field check only, not a format or length specification (none exists elsewhere in this spec).
26. The masked-identifier confirmation shall render in the same view as the connect action's success state — not a separate confirmation page requiring additional navigation.

### Felt "first win" moment

27. The system shall provide a read-only endpoint `GET /api/connections/:id/first-win`, tenant/ownership-scoped identically to the other `:id` connection routes (`401` if no session, `404` if the connection is not owned by the resolved tenant), returning `{ healthy: boolean, detail?: string, readyCount: number }` — any error thrown or returned by the underlying connector health check shall be mapped to a generic, safe `detail` string that never contains raw exception text, stack traces, or other internal state.
28. In the same view as the masked-identifier confirmation (requirement 26), the system shall display a "first win" signal for the newly-created connection consisting of: (a) the result of an immediate call to `GET /api/connections/:id/first-win` (requirement 27) for that connection — whose response normalizes any underlying connector error, thrown or returned, into a safe, non-throwing `{ healthy: false, detail }` shape, so the calling UI itself never observes a thrown exception regardless of what the underlying connector function (e.g. `checkConnectionHealth`) does — and (b) a count of the tenant's existing inventory items not yet present in `item_platforms` for that `platform`, framed as items ready to list on the newly-connected marketplace.
29. The "first win" signal shall be computed and rendered without requiring a separate page load or a distinct "view sync results" step from the connect confirmation — both shall appear together on first render after a successful connect.
30. If the tenant has zero inventory items available to list on the newly-connected platform (count is 0 per requirement 28b), the "first win" signal shall render the health-check result alone with copy appropriate to an empty inventory, rather than a failed or misleading "0 ready" framing.

## Non-functional requirements

- `GET /api/connections`, `GET /api/connections/:id/consent`, and `GET /api/disclosures/current` calls made to render the status view and connect cards must complete within [threshold TBD] at p95 for the page to be considered responsive — no numeric target exists in prior specs for read-path latency on these routes.
- No credential secret value shall ever be sent from the client back to itself for display (requirement 22 sources the masked identifier from pre-submission form state, not a round trip) or persisted in browser storage (`localStorage`/`sessionStorage`) beyond the lifetime of the single confirmation render — the masked identifier is a display convenience derived once, not a durable client-side cache of credential material.
- No submitted credential field value (secret or otherwise) shall appear in any client-side error banner, console log output, or rendered DOM at any point during this flow — including during error handling (e.g., a failed `POST /api/connections` call, a failed consent submission, or a `422` disclosure-version error), not only during the success/confirmation path described above.
- All new client-side fetches must include the existing session cookie (`SESSION_COOKIE_NAME`) and must not attempt to bypass `requireTenant`/`resolveOwnedConnection`'s tenant-scoping — this page introduces no new auth mechanism.
- The item-readiness count (requirement 28b) query must scope to the resolved tenant's own items only, consistent with existing tenant-isolation requirements in the multi-tenant foundation spec.

## Constraints

- Must consume `app/api/connections/route.ts`, `app/api/connections/[id]/route.ts`, `app/api/connections/[id]/consent/route.ts`, `app/api/connections/[id]/credential/route.ts`, `app/api/connections/[id]/reactivate/route.ts`, and `app/api/disclosures/current/route.ts` exactly as shipped — no modification to these routes' request/response shapes.
- No API route exists to trigger `createListing`/`updateListing`/`markSold`/`delist` from the UI layer (`lib/connectors/*` is server-only, called by nothing in `app/api` today) — the "first win" moment (requirements 27-28) is scoped to a non-mutating `checkConnectionHealth` call (wrapped by the new `first-win` endpoint) plus a read-only readiness count; it must not attempt to invoke connector write methods, since no route exists to do so safely within this increment's boundaries.
- No API route exists for an OAuth authorization-code redirect/callback for `eBay`/`Etsy`/`Amazon` — `POST /api/connections` accepts a `credential` object identically for all 8 platforms regardless of tier. The OAuth-tier connect cards use the same credential-submission contract as the credential-tier cards; only the disclosure copy and card grouping differ in this increment.
- `disclosure_versions.content` is one global, non-platform-scoped text field — this UX cannot fetch or display separate stored disclosure text per platform; per-platform risk framing is client-side copy layered over the shared fetched content (requirement 11).
- Must follow this repo's existing dark-mode class-based Tailwind convention (`dark:` variants paired with light classes, as in `SiteChrome.tsx`, `app/dashboard/page.tsx`, `app/inventory/page.tsx`) — no new theming system or CSS approach.
- Must integrate with `SiteChrome.tsx`'s global nav (add a `/connections` link) rather than introducing a separate nav pattern.
- Must not introduce a new endpoint for per-platform operability tier (real-Sandbox-tested / live-draft-only / inert / dry-run) if avoidable — this data is static per deployment (README table, driven by which env vars are configured) and may be hardcoded client-side as a tier map mirroring `README.md`, rather than requiring a new API route.
- `ConnectionMetadata` (the only shape `GET /api/connections`/`GET /api/connections/:id` return) has no field for a platform username/handle or any credential-derived identifier — the masked-identifier requirements (22-24) depend on capturing that value client-side at submission time, not on any existing or new server field.

## Out of scope

- Any change to `app/api/connections/**` or `app/api/disclosures/current` request/response contracts.
- Implementing a real OAuth authorization-code redirect/callback flow for eBay/Etsy/Amazon (registering redirect URIs, handling the callback, exchanging a code) — the current backend has no such route; building one is a separate increment.
- Triggering any real connector write action (`createListing`, `updateListing`, `markSold`, `delist`) from this UI — no route exists to call these safely from the client in this increment, and auto-selecting which inventory item to list without explicit per-item tenant choice is not built here.
- Cross-platform sync orchestration, price/inventory reconciliation, or any dashboard of ongoing listing health beyond the connection-status view itself.
- Visual design system changes, animation, or polish beyond making the states in this spec (empty state, connect cards, consent screen, status view, masked confirmation, first-win signal) legible within the repo's existing dark-mode/photo-forward conventions.
- Editing or managing the disclosure document's content/versions (an operator/admin concern, not exposed to tenants).
- Credential rotation UI beyond what's needed to reconnect a revoked connection (`PATCH /api/connections/:id/credential` exists but this spec does not require a dedicated "update my saved password" flow separate from reconnect).
- Notifications, email, or push alerts on connection-status changes (e.g., on suspension) — the status view is pull-based (tenant visits `/connections`), not push-based.
- Mobile-specific layout work beyond what the existing responsive conventions in `app/inventory/page.tsx`/`app/dashboard/page.tsx` already provide.

## Acceptance criteria

1. With zero connections for a tenant, `/connections` renders exactly one primary CTA and no per-platform cards, status rows, or competing actions.
2. After creating one connection, revisiting `/connections` no longer shows the zero-connections empty state.
3. `/connections` renders exactly 8 connect cards split into two groups (3 OAuth-tier, 5 credential-tier), matching `SUPPORTED_PLATFORMS` membership.
4. A connect card for a platform with an existing `active` or `suspended` connection has its connect action disabled; a card for a platform with no connection or a `revoked` connection has its connect action enabled.
5. Selecting an enabled connect card renders the risk-disclosure screen before any credential input field is shown, and before any `POST /api/connections` call is made (verified by network-call ordering in a test).
6. The consent screen offers no control that grants consent for more than one platform in a single interaction; each platform's consent is a separate `POST /api/connections/:id/consent` call.
7. No consent screen contains a pre-checked or default-accepted consent control — the affirmative action must be explicit.
8. Submitting consent with the current `disclosure_version` returns 201 and unblocks the credential step; submitting with a stale version surfaces the `stale_disclosure_version` error without silently proceeding.
9. The status view renders `active` in green, `suspended` in yellow, `revoked` in red, for connections in each of those states respectively, and renders at least one blue-coded informational element (consent-staleness or operability tier) that is never used for the three enum states.
10. A connection with `status: active` but no valid current consent (per `GET /api/connections/:id/consent`) displays both facts distinctly — the status color reflects `active`, and a separate, visibly distinct indicator reflects invalid/stale consent.
11. Triggering `POST /api/connections/:id/reactivate` on a `suspended` connection from the status view transitions its displayed status to `active` on the next render; attempting it on an `active` or `revoked` connection surfaces the `409 not_suspended` error visibly rather than failing silently.
12. Immediately following a successful connect (`201` from `POST /api/connections`), the same rendered view shows a masked identifier (e.g. `connected as @h***e`) derived from the just-submitted form value, without any additional network round trip carrying that value back from the server.
13. Immediately following that same successful connect, the same rendered view (no additional navigation) shows both a `checkConnectionHealth` result and a count of the tenant's items not yet listed on that platform.
14. When the item-readiness count is 0, the first-win signal still renders the health-check result and does not present a negative/failed framing for the 0 count.
15. No network request or DOM output produced by this feature contains a raw credential/secret value at any point — verified by a test asserting the submitted secret field's value never appears in any subsequent API response body or rendered masked-identifier string (only the non-secret identifier field may appear, masked).
16. `npm test`/`npm run test:e2e` covering this feature exercise all of the above using the existing scratch-DB test isolation (`BOOKSELLER_DB_PATH`), with no live third-party credentials required, consistent with the connector tier's "wired but inert until credentialed" posture.
